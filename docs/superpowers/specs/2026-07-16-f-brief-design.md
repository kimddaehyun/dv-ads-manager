# F-Brief - 광고주 보고 문구 생성 (AX 1호)

**작성일**: 2026-07-16
**상태**: 설계 승인 대기
**전제**: 사내 전용 (디브이마케팅 AE). 외부 배포 없음.

---

## 1. 배경

AE는 매주/매월 광고주에게 카카오톡으로 성과를 보고한다. 실제 보고 로그 5건을 분석한 결과,
5명의 AE가 쓴 문구가 동일한 뼈대를 갖고 있었다.

| 블록 | 실제 예시 | 정체 |
|------|-----------|------|
| ① 인사 | `안녕하세요:)` / `안녕하십니까 대표님!` | AE별 고정 |
| ② 기간 + 3지표 | `지난 30일 동안 검색광고, GFA 포함` `▶광고비 : 267,558원` `▶전환매출액 : 1,660,420원` `▶광고수익률 : 620.58%` | 순수 데이터 |
| ③ 전기 대비 | `매출은 약 34만 원 감소하였으며, 수익률 또한 862% > 621%로 하락` | 순수 데이터 |
| ④ 원인 진단 | `객단가가 높은 [온열 찜질기] 상품에서 전환이 발생하지 않아` | 데이터가 후보 제시 + AE 판단 |
| ⑤ 액션 | `입찰가 유지하며 추이 확인 후 조정하겠습니다` / `OFF 하여도 괜찮을지 문의드립니다` | AE 판단 |

**문단마다 "사진"이 1:1로 짝을 이룬다.** 사진은 광고관리자 화면 캡처 또는 엑셀에 손으로 색칠한 표다.

### 관찰 1 - 문구의 절반은 AI가 필요 없다

①②③은 전부 F-Report가 이미 가진 숫자다. `ReportModel.totalCurrent` / `totalPrev`가 설정 기간과
이전 동일 기간을 모두 담고 있어 `862% > 621%`는 지금 당장 계산된다.

### 관찰 2 - ④의 문장들은 전부 임계값 쿼리다

- `광고비 1만 원 이상 소진되었으나 전환이 발생하지 않은 '대나무돗자리' 키워드` → `cost >= 10000 AND conv == 0`
- `[네이버 메인] 지면에서 전환이 발생하지 않아` → 지면별 동일 규칙
- `초록색칠된 키워드는 800% 이상 / 노란색칠은 600%대 / 색칠 안 된 건 목표 미달` → ROAS 구간 분류

임유엽 AE는 **엑셀을 손으로 색칠하고 있다.** 그 색칠이 규칙 그 자체다.

이는 `docs/ROADMAP.md` Task 013 보류 사유에 이미 기록된 내용과 일치한다:
> AE 광고주 보고 카톡 분석(2026-05-19) 결과 진짜 페인은 "ROI 임계값 기반 자동 분류 + 일괄 입찰 변경"

### 관찰 3 - 순위 + 효율 결합이 우리만의 무기다

`전환이 발생한 주요 키워드들의 노출 순위가 2페이지로 다소 낮게 확인되어` - 이 판단에는
**효율(F-Report)과 순위(F001 `estimateRank`)가 동시에** 필요하다. 두 데이터가 한 확장 안에 있는 건
우리뿐이다. 네이버 광고관리자도 이 조합을 제공하지 않는다.

## 2. 무엇을 만드는가

**F-Brief**: 계정 하나를 골라 기간을 지정하면, 카톡에 그대로 붙여넣을 수 있는
**보고 문구 + 색칠된 분석 표 이미지**를 블록 단위로 생성한다.

F-Report(엑셀)와 **독립된 기능**이다. 리포트를 뽑든 안 뽑든 따로 쓴다. 데이터 수집 계층만 공유한다.

> **AX**는 이 프로젝트의 AI 도입 이니셔티브 전체를 가리키는 우산 용어다. F-Brief는 그 1호다.
> 후속 후보는 §11 참조.

### 범위 안

- 계정 **1개** 대상 문구 생성
- 규칙 엔진 기반 후보 추출 + AE 선택
- AI 문장 조립 (사내 서버 중계)
- 분석 표 **이미지 생성** (캡처 아님) + 클립보드 복사
- 계정별 목표 ROAS 저장

### 범위 밖 (YAGNI)

- **다계정 일괄 생성** - 문구는 광고주마다 내용이 다르고 AE가 하나씩 읽어야 한다. 1개가 검증되면 재검토.
- **실제 대시보드 화면 캡처** (`chrome.tabs.captureVisibleTab`) - §10 참조.
- **입찰가 변경 실행 + 액션 기록 DB** - 1단계에서 액션은 문장으로만 선언. 실행은 기존 F001 팝오버 경로.
  단, **2단계로 방향 확정** - 항목별 실행 버튼 + 조정 기록 DB + 보류함 (§15).
- **결제/계정/사용량 제한** - 사내 전용이므로 불필요.
- 엑셀 리포트(.xlsx) 변경 - 양식 바이너리를 건드리지 않는다.

## 3. 핵심 원칙 - AI가 손댈 수 있는 범위를 좁힌다

LLM은 규칙을 **실행**하지 않는다. 확률적으로 글을 쓴다. "숫자를 바꾸지 마라"고 지시해도 가끔 바꾼다.
프롬프트로 없앨 수 없는 성질이다.

따라서 이 설계는 AI에게 규칙 준수를 **부탁하지 않고**, 어길 수 없는 상태에 둔다.

### 3겹 방어

**1겹 - 안 물어본다.** ①②③ 블록은 AI를 거치지 않는다. 코드가 문자열로 조립한다.
```ts
"▶광고비 : " + cost.toLocaleString() + "원"
```
AI가 이 숫자를 틀릴 확률은 0이다. 물어본 적이 없다. **문구의 절반이 여기서 안전해진다.**

**2겹 - 재료를 안 준다.** 리포트 전체를 보내지 않는다. **체크된 사실만** 보낸다.
체크 안 된 후보는 프롬프트에 존재하지 않으므로 AI가 그것을 언급할 수 없다.

**3겹 - 검산한다.** AI가 쓴 문장에서 숫자를 추출해 우리가 준 값 집합과 대조한다.
없는 숫자가 나오면 해당 블록에 경고를 띄운다. 코드가 하는 일이라 확실하다.

### 남는 구멍과 그 처리

3겹으로도 못 막는 것이 있다. AI가 **일반 상식으로** 끼워 넣는 문장이다:

> "여름 시즌 특성상 관련 수요가 증가하는 시기이므로..."

우리가 준 적 없고, 틀린 말도 아니고, 숫자가 아니라 검산에 안 걸린다.

**따라서 AI 판단이 들어간 문단은 시각적으로 표시한다** (§8 좌측 주황 선). 기술로 막을 수 없는
구간을 AE 눈에 띄게 만드는 것이 마지막 방어선이다. **최종 방아쇠는 항상 사람이 당긴다** -
AE가 읽고, AE가 복사 버튼을 누르고, AE가 카톡에 붙여넣는다.

> **이 기능에서 AI의 역할은 "분석가"가 아니라 "번역기"다.**
> 분석은 규칙 엔진이, 판단은 AE가, AI는 문장 조립만 한다.

### 4.1 `collectReportData()` 추출 (필요한 만큼의 정리)

현재 `buildReportBytes`(`report-build.ts:640-800`, 약 160줄)는 **데이터 수집과 엑셀 생성을 섞어서**
한다. 한 함수 안에서:

1. 양식 `.xlsx` fetch + `openXlsx` *(엑셀 전용)*
2. `fetchBrandContracts` → 일할 계산 *(수집)*
3. `fetchGfaData` *(수집)*
4. `fetchAdvancedReport` x3 (파워링크/쇼핑 키워드, 상품) *(수집)*
5. `buildReportModel` / `buildKeywordGroups` / `buildProductRows` *(가공)*
6. `fill*` / `render*` *(엑셀 전용)*

F-Brief는 2~5만 필요하고 1·6은 필요 없다. 따라서 2~5를 `collectReportData()`로 추출한다.

```ts
export interface ReportData {
  model: ReportModel;           // 종합/검색/디스플레이 합계 + byDay/byPlacement/byGender/byAge
  searchTypes: NamedMetrics[];
  displayData: GfaData;
  campGroups: CampaignGroup[];
  plKeywords: KeywordGroup[];   // 파워링크 키워드
  shKeywords: KeywordGroup[];   // 쇼핑검색 키워드
  shProducts: ProductRow[];     // 쇼핑검색 상품

export async function collectReportData(
  target: ReportTarget, range: DateRange, meta: ReportMeta,
): Promise<ReportData>;
```

- `buildReportBytes` = `collectReportData()` + 엑셀 생성. **동작 불변.**
- `buildBriefData` = `collectReportData()` + 규칙 엔진.

**검증**: 추출 전후로 동일 계정·동일 기간의 `.xlsx`를 생성해 바이트 비교. 차이가 있으면 실패.

**주의**: 기존 병렬 구조(`brandP`/`displayDataP`를 await로 막지 않고 promise로 먼저 출발시키는 패턴,
`report-build.ts:648-660` 주석 참조)를 그대로 보존한다. 성능 감사 2026-07-02에서 확정된 구조다.

## 5. 규칙 엔진 (`brief-rules.ts`)

AI 없음. 순수 함수. 입력 `ReportData` + 목표 ROAS → 출력 `BriefCandidate[]`.

```ts
export type BriefAction =
  | "raise" | "hold" | "lower" | "exclude" | "ask" | "custom";

export interface BriefCandidate {
  kind: BriefKind;
  /** 문구에 들어갈 사실. AI에게 이것만 전달된다. */
  facts: Record<string, string | number>;
  /** 딸려나올 표. kind가 결정한다. AE가 고르지 않는다. */
  table: BriefTableSpec;
  /** 기본 선택 여부. 완전자동 모드에서 AI가 이 값을 덮어쓸 수 있다. */
  selected: boolean;
  action?: BriefAction;
  actionText?: string;  // action === "custom"
}
```

### 후보 종류와 표의 대응

표 종류는 **후보 종류가 결정한다.** AE가 따로 고르지 않는다.

| `BriefKind` | 조건 | 딸려나오는 표 |
|---|---|---|
| `zeroConvKeyword` | `cost >= 임계` AND `purchaseConv == 0` | 키워드별 효율 (색칠) |
| `highRoasLowRank` | 초록 구간 AND `estimatedRank >= 저순위 임계` | 키워드별 효율 + 순위 |
| `belowTargetKeyword` | `cost >= 임계` AND **무색 구간** (`purchaseConv > 0`) | 키워드별 효율 (색칠) |
| `zeroConvPlacement` | 지면 `cost >= 임계` AND `conv == 0` | 지면별 효율 |
| `productConvDrop` | 전기 대비 `purchaseConv` 감소 AND 매출 기여 상위 | 상품별 효율 (전기 대비) |

### ROAS 구간 (단일 정의)

색칠과 후보 조건이 **같은 구간 정의를 쓴다.** 두 곳에 따로 쓰면 어긋난다.

```ts
type RoasBand = "green" | "yellow" | "none";

function roasBand(roas: number, target: number): RoasBand {
  if (roas >= target) return "green";        // 상향 여지
  if (roas >= target * 0.75) return "yellow"; // 유지
  return "none";                              // 하향
}
```

임유엽 AE 로그가 이 3단을 그대로 서술한다 (목표 800% 기준):
- 초록 `800% 이상 ... 입찰가 상향` → `green`
- 노랑 `1위노출이 발생하거나 600%대 ... 입찰가 유지` → `yellow` (600/800 = 0.75)
- 무색 `목표수익률을 달성하지 못하였기에 입찰가 하향` → `none`

**`belowTargetKeyword`가 `roas < 목표`가 아니라 `none` 구간인 이유**: `roas < 목표`로 잡으면
노랑(유지 대상)까지 "목표 미달"로 딸려 들어간다. AE는 노랑을 하향하지 않는다. 액션이 다르면
같은 후보가 아니다.

`purchaseConv > 0` 조건은 `zeroConvKeyword`와의 중복을 막는다. 전환 0인 키워드는 ROAS도 0이라
자동으로 `none`에 떨어지는데, 그건 이미 별도 후보다.

### 임계값

| 값 | 기본 | 근거 |
|---|---|---|
| 비용 임계 | 10,000원 | 보고 로그 5건 중 4건이 "1만원 이상" 문구 사용 |
| 목표 ROAS | **계정별 저장** (§7) | 로그에 300%/600%/800% 등장. 계정마다 다름 |
| 노랑 하한 | 목표 x 0.75 | 위 역산 |
| 저순위 | `rank >= 6` | 로그의 "2페이지" = PC 기준 11위~. MVP는 보수적으로 6위 (§14 미해결) |

**목표 ROAS가 없으면** `belowTargetKeyword` / `highRoasLowRank` / 색칠은 **후보에서 제외**하고,
패널 상단에 "목표 수익률을 설정하면 키워드 분류를 제안합니다" 안내 + 설정 링크를 띄운다.
자동 추정(계정 평균)은 **하지 않는다** - 계정이 통째로 부진하면 전부 "정상"으로 나와 무용지물이 된다.

### 조정 레버 확장 카탈로그 (2026-07-16 웹 조사 반영, 범위: 전부)

위 5종은 1단계 MVP다. 네이버 검색광고에서 대행사가 **실제로 조정하는 레버**를 조사해(출처 하단),
"조정 가능한 축 전부"를 후보로 확장하기로 확정했다(2026-07-16 결정). 골격(Task 4~6) 완성 후
아래 순서대로 얹는다. **상세 규칙·임계는 각 Task 구현 시점에 확정** - 여기서는 목록과 원칙만 못박는다.

**확장의 전제 3원칙:**

1. **계층이 액션을 정한다.** 조정 레버는 계층마다 붙는 곳이 다르다. 규칙은 레버가 사는 계층에서 후보를 낸다.
   - 총평(3지표) = 계정 · 키워드 입찰 = 키워드 · 상품/소재 = 소재 · 예산 = 광고그룹 · 타게팅(성별·연령·시간·요일·지역) = 그룹 타게팅
2. **많아도 된다, 단 정렬 + 문턱 + 기본해제가 조건.** AE가 최종 필터이므로 후보는 많을수록 놓칠 위험이 준다.
   그러나 (a) 각 후보는 비용/매출 문턱을 넘어야 하고(잡음 차단), (b) 영향 순 정렬(강한 것 위로),
   (c) 기본 전부 해제(`selected: false`)여야 한다. 이 셋이 없으면 "많음"은 그대로 소음이 된다.
3. **데이터 확보가 병목이다.** 규칙 추가는 값싸고 새 수집이 비싸다. 아래 tier는 데이터 확보 난이도 순.

**확장 후보 (`BriefKind` 추가분):**

**총평은 계정, 진단은 계층별:** 확장해도 보고서 맨 앞 3지표는 계정 전체 합계(검색+GFA)로 둔다.

## 6. AI 조립 (`brief-compose.ts` + 사내 서버)

### 6.1 왜 서버인가

사내 전용이므로 사용자 본인 키 방식도 가능하지만, 서버 중계를 택한 이유:

- **프롬프트를 확장 재배포 없이 튜닝** - 문구 품질은 프롬프트 반복으로만 올라간다. 확장에 박으면
  매번 빌드 + 사내 재배포 + AE들이 chrome://extensions Reload를 해야 한다.
- API 키를 AE에게 노출하지 않음.
- 말투 샘플(실제 광고주 보고 로그)이 서버에만 있음.

**PRD 원칙 조정 필요**: `docs/PRD.md`의 "사용자 데이터 외부 전송 0건"은
**"광고 데이터는 네이버와 사내 서버 외로 나가지 않는다"**로 개정한다 (§12).

### 6.2 서버 - Supabase Edge Function

```
POST https://<project>.supabase.co/functions/v1/brief-compose
Authorization: Bearer <사내 AE 토큰>

요청:  { advertiser, periodText, totals, prevTotals, facts[], memo }
응답:  { blocks: [{ text, isAiJudgment }] }
```

- **Edge Function 1개.** 1단계에서는 DB 테이블 없음, Auth 없음, Storage 없음.
  (2단계에서 액션 기록 테이블이 추가된다 - §15. 그때도 Supabase Auth는 쓰지 않는다.)
- 인증: 사내 AE에게 발급한 토큰의 **화이트리스트 대조**(Edge Function 환경변수).
  Supabase Auth를 쓰지 않는다 - 사내 전용이라 계정 시스템이 과하다.
- 토큰은 옵션 페이지에 입력, `chrome.storage.local`에 보관.
- AI API 키는 Edge Function 환경변수. 확장에 없다.
- 프롬프트·말투 샘플도 Edge Function 안에 둔다 → 확장 재배포 없이 튜닝 (§6.1).
- 저장 없음. 요청/응답을 로그에 남기지 않는다 (광고주 데이터).

**`@supabase/supabase-js`를 다시 넣지 않는다.** Edge Function 호출은 단순 HTTPS POST라
`fetch`로 충분하다. SDK는 DB/Auth/Realtime을 위한 것이고 우리는 아무것도 안 쓴다.
2026-05-15에 제거한 의존성을 번들 크기만 늘리며 되살릴 이유가 없다.

**되돌리는 것** (2026-05-15 라이선스 제거 시 삭제됨):
- `manifest.config.ts` `host_permissions`에 `https://<project>.supabase.co/*` 1개 추가.
- Supabase 프로젝트 자체 - 2026-05-15 "인프라 분리 결정"대로 **본 확장 전용 프로젝트**를 쓴다.

**되돌리지 않는 것**: `license.ts` / `supabase.ts` / `LicenseUi` / RPC `verify_access`.
F-Brief는 라이선스와 무관하다. 사내 전용이므로 접근 제어는 토큰 화이트리스트로 끝난다.

### 6.3 프롬프트 구성

```
너는 디브이마케팅 AE다. 아래 말투로 광고주 보고 문구를 써라.

[말투 샘플]
(실제 보고 로그 4~5건을 원문 그대로)

[이번 데이터]
광고비 267,558원 / 전환매출 1,660,420원 / ROAS 620.58%
이전 기간 ROAS 862.53%

[말할 것]
- 온열 찜질기 상품 전환 0건, 매출 -34만원 → 액션: 입찰가 유지 후 추이 확인
- AE 메모: 6월 말부터 아이패드 CPC 낮춰 운영 중

[규칙]
위에 준 사실 외에는 쓰지 마라. 숫자를 바꾸지 마라.
```

**말투 샘플을 넣는 것이 핵심이다.** "간결하고 정중하게 써라"라고 *설명*하는 것보다 실제 문구를
*보여주는* 쪽이 압도적으로 잘 맞는다. `안녕하세요:)`의 이모티콘까지 복제된다.

### 6.4 두 모드

내부적으로 같은 엔진이다. **완전자동은 AE선택의 체크박스를 AI가 미리 채운 상태**일 뿐이다.
구현이 두 배가 되지 않는다.

| 모드 | 진입 | 동작 |
|---|---|---|
| **완전자동** (기본) | 패널 열면 바로 | AI가 후보 선정 + 액션 판단까지. AE는 읽고 복사. |
| **AE선택** | "직접 고르기" 버튼 | 후보 목록에서 AE가 체크 + 액션 지정 → 문구 생성 |

완전자동에서 AI가 고른 액션은 **반드시 좌측 주황 선으로 표시**된다 (§8).

**완전자동의 위험을 명시한다**: 실전에서 무서운 것은 오작동이 아니라 **익숙해지는 것**이다.
몇 주 잘 맞으면 AE가 스킴하고 붙여넣기 시작한다. 그때 광고주에게 나간 약속은 대행사가
이행해야 한다. 주황 선은 이에 대한 유일한 방어다.

### 6.5 검산

응답의 각 블록에서 숫자를 정규식으로 추출 → 우리가 보낸 `facts` + `totals`의 값 집합과 대조.
집합에 없는 숫자가 있으면 그 블록에 경고 배지. **차단하지 않고 표시만 한다** (오탐 가능 -
"30일", "2페이지" 같은 정상 숫자가 있다). 판단은 AE에게 맡긴다.

## 9. 분석 표 이미지 (`brief-table.ts`)

**캡처가 아니라 생성한다.** 표의 데이터를 우리가 이미 갖고 있다.

`<canvas>`에 표를 그리고 ROAS 구간별로 행 배경을 칠한 뒤 `canvas.toBlob()` → PNG →
`navigator.clipboard.write(new ClipboardItem({"image/png": blob}))`. AE는 카톡에서 `Ctrl+V`.

### 왜 캡처보다 나은가

| | 캡처 | 생성 |
|---|---|---|
| 긴 표 | 스크롤 밖은 잘림 | 전부 나옴 |
| DOM 셀렉터 | 네이버가 바꾸면 깨짐 | 무관 |
| 색칠 | AE가 엑셀에서 손으로 | 자동 |
| 해당 페이지 | 열려 있어야 함 | 불필요 |

**AE가 엑셀 열어서 손으로 칠하던 작업이 통째로 사라진다.**

### 사양

- 색칠: 초록 `#16a34a`/10, 노랑 `#d97706`/10, 무색. (DESIGN.md `Semantic (State Colors)` 재사용)

## 11. AX 후속 후보 (이번 범위 밖, 참고용)

AI가 **진짜로 필요한 곳**(규칙으로 불가능한 곳)만 골랐다.

**1순위 - 확장소재 카피 생성 (F-AssetBulk)**
AE가 추가제목·추가설명·홍보문구를 손으로 타이핑한다. 그런데 `product-page-scrape.ts`가 이미
상품 페이지에서 제목·이미지를 긁어온다. 상품 정보 → "15자 이내 추가제목 5개". 글자수 제한이
명확해 검증이 쉽고, 텍스트가 짧아 비용이 거의 없다. **규칙으로 절대 못 하는, 순수하게 AI가
잘하는 일** - F-Brief보다 AI 적합도가 높다.

**2순위 - 검색어 → 제외 키워드 후보**
리포트가 `expKeyword`(실제 검색어)를 이미 수집한다. "우리 상품과 무관한데 돈만 쓰는 검색어"를
골라내는 것은 의미 판단이라 임계값으로 불가능하다. 비용 발생 검색어로 좁히면 양도 적다.

**후보에서 제외한 것**
- F-ChangeWatch 변경이력 요약 - "설정 N개 변경"을 문장화하는 것은 가치 대비 소음.
- 리포트 이상 탐지 - 규칙으로 충분. AI 낭비.
- 세팅안 초안 생성 - F-Setup은 기존 계정을 뽑는 도구지 만드는 도구가 아니다. 스코프 불일치.

## 13. 검증

Chrome 확장은 자동 E2E보다 실제 페이지 수동 검증이 안정적이다 (ROADMAP 개발 워크플로우).

### 자동

| 대상 | 방법 |
|---|---|
| `collectReportData()` 추출 | 추출 전후 동일 계정·기간의 `.xlsx` 바이트 비교. 차이 = 실패 |
| `roasBand` | 경계값 표. `roas == 목표` → green, `roas == 목표 x 0.75` → yellow, 한 단위 아래 → none |
| 규칙 엔진 | 순수 함수. 고정 `ReportData` → 기대 `BriefCandidate[]`. 경계값(`cost == 10000`) 포함. **노랑 구간 키워드가 `belowTargetKeyword`에 안 들어가는지** |
| 검산 로직 | 조작된 응답("267,559원")을 넣어 경고가 뜨는지 |
| 표 렌더 | 0행/1행/20행/21행(잘림) 스냅샷 |

### 수동 (필수 체크리스트)

- [ ] 목표 ROAS 미설정 계정 - 분류 후보 없이 안내 + 설정 링크
- [ ] 검색광고 자격증명 미등록 - `highRoasLowRank`만 빠지고 나머지 정상
- [ ] 디스플레이(GFA) 없는 계정 - "검색광고, GFA 포함" 문구가 "검색광고"로
- [ ] 전환 0 계정 - ROAS 나눗셈 0 분모
- [ ] 서버 응답 실패 - 규칙 엔진 결과(후보 목록 + 표)는 여전히 보임
- [ ] 표 이미지 클립보드 → 실제 카카오톡 붙여넣기
- [ ] 완전자동 문구의 액션 문단에 주황 선이 실제로 보임
- [ ] 복사한 텍스트에 주황 선/서식이 안 딸려감

### 성공 기준

**AE가 생성된 문구를 거의 그대로 카톡에 붙여넣는다.**

실패 모드는 명확하다 - "그럴듯한데 우리 말투가 아니라서 결국 다 다시 씀". 그러면 시간을
아끼는 게 아니라 오히려 쓴다. 첫 사용자는 실제 보고 로그를 제공한 AE로 한다.

## 15. 2단계 - 실행 버튼 + 액션 기록 DB (방향 확정 2026-07-16, 이번 구현 범위 밖)

Task 1~11(문구 생성) 완성 후 착수한다. 실행은 실제 입찰가를 바꾸는 일이라 문구 생성이
검증된 뒤에 얹는다.

### 확정된 흐름

1. **데이터로 개선 후보 추출** (규칙 엔진, 1단계 그대로)
2. **AE가 후보 선택** → 카톡 문구 생성 (1단계 그대로)
3. **항목별 실행 버튼** - 후보 선택 화면의 각 항목 옆에 실행 버튼. 문구 생성과 실행은
   같은 화면에 있지만 서로 묶이지 않는다 (일괄 자동 실행 금지).
   - **즉시 실행형** ("입찰가를 조정하겠습니다") - 카톡 생성 직후 AE가 눌러 바로 실행.
   - **문의형** ("OFF 하여도 괜찮을지 문의드립니다") - 광고주 답장이 필요하므로 **보류함**에
     저장. 답이 오면 패널을 다시 열어 "대기 중인 액션 N건"에서 실행. 패널을 켜놓고
     기다리지 않는다.
4. **실행 순간 조정 기록을 DB에 자동 저장** - 키워드, 날짜, 이전 입찰가 → 새 입찰가, AE, 사유.

### 왜 로컬(chrome.storage)이 아니라 DB인가

- **팀 가시성** - AE 20명 규모. 보류 건수·방치 건을 팀 차원에서 볼 수 있다.
- **인수인계·PC 교체·재설치 유실** 문제가 사라진다.
- **"지난 액션 추적" 규칙의 재료** - 이게 핵심. 다음 보고 생성 시 규칙 엔진이 조정 기록을
  읽어 "지난주 입찰가를 조정한 [키워드]는 조정 후 수익률 412% → 688%로 개선" 같은 후보를
  자동 생성한다. 제안 → 카톡 → 실행 → 다음 주 결과 보고의 루프가 닫힌다.
- 보고 문구 발송 이력도 저장하면 "지난 보고에서 약속한 건"을 다음 후보로 띄울 수 있다.
- **계정 설정(`targetRoas` 등 `MultiAccountUserMeta`)의 DB 이관도 이때 검토** - 현재는
  브라우저 로컬이라 AE마다 따로 입력해야 한다. DB로 올리면 팀장이 계정별 목표를 한 번
  정하면 모든 AE에게 같은 기준이 적용된다.

### 데이터 분담 (공식 API 사용 불가 전제)

계정의 검색광고 공식 API는 쓸 수 없다(개인 API는 검색량 조회용뿐). 성과 데이터는 항상
**AE가 로그인한 브라우저 안에서 확장이** 내부 통신으로 그때그때 조회한다 (F-Report와 동일).

- **DB에는 성과 숫자를 저장하지 않는다.** 행동 기록(무엇을 언제 얼마로 바꿨는지)만 저장.
- "조정 후 성과" 문장 = DB의 조정 기록 + 패널을 연 순간 확장이 가져온 성과를
  **브라우저 안에서 결합**해 생성.
- 서버 측 자동 수집·예약 분석은 불가능하고, 필요하지도 않다 (패널은 AE가 열어서 쓰는 도구).

### 보안 원칙의 의도적 조정

"광고주 데이터 외부 전송 0건" 원칙을 **의도적으로 한 단계 더 연다**: 키워드명·입찰가가
사내 Supabase DB에 저장된다 (§6.1의 "중계만, 저장 없음"에서 확장). 사내 전용 인프라이므로
수용. 2단계 착수 시 PRD·CLAUDE.md에 명시 개정.

### AE 식별

로그인 시스템은 만들지 않는다. 옵션 페이지에서 AE 이름을 1회 선택 → 기록에 함께 저장.
기존 토큰 화이트리스트(§6.2)에 AE 이름을 매핑하는 것으로 충분할 수 있다 - 2단계 설계 시 확정.

### 비용 (AE 20명, 광고주 ~1,000, 월 ~4,000건 보고 기준)

- **Supabase 무료 플랜으로 충분.** Edge Function 호출 월 ~1만 회(한도 50만), 전송량 무시 수준.
- DB 500MB 한도만 장기 관리 필요 - 조정 기록 월 10~20MB 추정. **1년 경과 기록은 삭제 또는
  요약 보관** 규칙으로 무료 유지. "지난주 조정 참고" 용도는 최근 몇 달이면 충분.
- 무료 플랜은 7일 무요청 시 일시정지되나, 이 규모면 매일 사용되므로 사실상 무관.
  연휴 대비 GitHub Actions 주 2회 ping을 걸어두면 안전.
- 고정비는 Gemini 사용료뿐 (Flash-Lite, 월 4천 건 기준 몇천 원~1만 원 안쪽).
