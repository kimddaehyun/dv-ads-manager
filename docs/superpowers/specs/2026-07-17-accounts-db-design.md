# F-Accounts — 회원 체계 + 서버 DB 설계

2026-07-17 브레인스토밍 확정. 로컬(chrome.storage) 중심 구조를 회원 계정 + Supabase DB 중심으로 전환한다.

## 1. 배경과 목표

지금은 자격증명·계정 설정·그룹이 각 PC의 `chrome.storage.local`에만 있다. 회원 체계를 도입해:

- AE가 어느 PC에서든 로그인만 하면 자기 설정이 따라온다.
- 온보딩이 "가입 → 관리자 승인"으로 끝난다 (자격증명·코드 수동 배포 폐지).
- 보고 이력이 서버에 남아 "지난 조치 → 이번 성과" 비교 보고가 가능해진다 (2단계).

**PRD 참고**: 기존 PRD의 "광고 데이터 외부 저장 금지" 원칙은 이 설계로 개정 대상이다(PRD는 새로 작성 예정). 광고 데이터 저장은 문제가 아니며, 판단 기준은 필요뿐이다. 성과 원본은 네이버에서 실시간 조회로 충분하니 저장하지 않고, **다시 만들 수 없는 기록(보고 이력)** 만 저장한다.

## 2. 확정 요구사항

| 결정 | 내용 |
|---|---|
| 데이터 소유 | **개인 소유** — 회원마다 자기 자격증명·그룹·설정. 공유 없음 |
| 가입 | **누구나 가입 + 관리자 승인제** (대기/승인/차단) |
| 미승인/미로그인 | **전부 잠금** — 확장 기능 전체가 "로그인이 필요해요" 안내만 |
| 관리자 화면 | **확장 옵션 페이지 안** 관리자 탭 (관리자 로그인 시만 노출) |
| 기존 데이터 | 첫 로그인 때 로컬 → 서버 자동 이사 |
| brief_token | 폐기 — 로그인 세션 토큰으로 대체 |
| 접근법 | **Supabase Auth + DB 직접 접근(RLS) + `@supabase/supabase-js`** |

SDK 재도입 근거: 2026-05-15 제거는 라이선스 확인 하나 때문이었다. 지금은 Auth(토큰 갱신·세션 복원·비밀번호 재설정) + DB + 관리자까지 쓰므로 직접 짤 보안 코드를 없애는 실익이 번들 증가(+100~200KB)보다 크다.

검토 후 기각한 대안: ② REST 직접 호출(토큰 갱신·세션 복원을 손으로 — 잔버그 원천), ③ 전부 Edge Function 경유(CRUD마다 서버 코드 — 사내 규모에 과함).

## 3. DB 테이블 (프로젝트 `gvyvrjncpwmcwycebrhf` 그대로 사용)

모든 테이블에 RLS "본인 행만 접근" + **승인된 회원만** 정책. 미승인은 읽어도 빈 결과.

1. **`profiles`** — 가입 시 자동 생성. 이메일, 표시 이름, 승인 상태(`pending`/`approved`/`blocked`), 관리자 여부(`is_admin`). 관리자는 전체 조회 + 상태 변경 가능(별도 정책).
2. **`credentials`** — 네이버 검색광고 API 자격증명, 1인 1벌. Customer ID, Access License, **Secret Key는 암호화 저장**(§6).
3. **`account_meta`** — 광고계정별 설정. 지금의 `MultiAccountUserMeta` 필드 그대로(별칭·즐겨찾기·비즈머니/브랜드 임계값·changeWatch·targetRoas) + "내 계정 목록 추가 여부"(현 `multi_account_added_list` 흡수).
4. **`account_groups`** — 계정 그룹. 지금의 `MultiAccountGroup`(이름/순서/accountNos) 그대로.
5. **`brief_history`** (2단계) — 보고 이력. §7.

**DB로 옮기지 않는 것**: 스냅샷·디렉터리 캐시(성능용, 로컬 유지), 광고 성과 원본(실시간 조회로 충분).

## 4. 인증 흐름

- **가입/로그인**: 이메일+비밀번호, 옵션 페이지에서. 가입 직후 "승인 대기 중" 화면.
- **잠금**: 로그인+승인 아니면 오버레이·대시보드·팝업 전부 안내만. 상태 확인은 시작 시 1회 + 캐시.
- **세션**: SDK가 자동 갱신. 세션은 `chrome.storage` 보관(SDK storage adapter) — 브라우저 재시작에도 유지.
- **brief-compose 인증**: Authorization에 로그인 JWT → 서버 함수가 검증 + `profiles.status = approved` 확인. `BRIEF_TOKENS` 화이트리스트 제거.

## 5. 마이그레이션 (로컬 → 서버)

- 로그인 성공 직후 1회. 자격증명·account_meta·groups·추가 목록을 업로드.
- **충돌 규칙: 서버가 이긴다.** 서버가 비어 있으면 로컬 → 서버 전부 복사, 이미 있으면 로컬을 서버로 덮음. 병합 없음(주 사용 PC 1대가 일반적 — 복잡도 대비 실익 없음).
- 이사 후 로컬은 캐시: 쓰기는 서버 먼저 → 성공 시 로컬 갱신. 네트워크 단절 시 읽기는 캐시, 쓰기는 "저장 안 됐어요" 안내.
- 이사 완료 시 `brief_token` 삭제 + 옵션 페이지 이용 코드 카드 제거.

## 6. 보안

- RLS가 1차 방어 — 확장 코드 버그로도 남의 데이터 접근 불가.
- **Secret Key 암호화**: 저장·조회를 Edge Function 경유로만. 서버 환경변수 열쇠로 암호화해 DB에 저장 — DB 콘솔에서도 원문 불가시. 로그인한 본인에게만 복호화해 반환(확장은 로컬 캐시에 보관해 네이버 API 호출에 사용).
- 관리자 지정 bootstrap: 최초 1회만 Supabase 콘솔에서. 이후 관리자 화면에서 지정 가능.

## 7. 보고 이력 (`brief_history`, 2단계)

**저장은 원본 구조로, 소비는 그때그때 변환** — LLM용 형식으로 미리 가공해 저장하지 않는다(프롬프트가 바뀌면 과거 이력이 구형식이 됨). 같은 구조를 AE 화면(목록·표)과 LLM facts(코드가 비교 계산 후 숫자만) 두 갈래로 렌더링.

저장 시점 = **AE가 문구를 복사한 순간** (생성만 하고 닫으면 기록 없음). 내용 3종:

1. 보낸 문구 전문
2. **조치 내역(구조화 JSON)** — 선택된 후보의 kind/facts/action ("키워드 A·B 하향" 등)
3. 그 시점의 성과 스냅샷 — 기간 합계 + 조치 대상들의 당시 지표

이것으로 규칙 엔진에 **"지난 조치 추적" 후보**가 생긴다: "지난 보고(7/10)에서 하향한 키워드 3개 → 이번 광고비 -32%, 수익률 410%→640%".

**보고 유형 카탈로그** (사용자 확정, 조합 가능): ①수정했다고 통보 ②이래서 이렇게 수정하려 한다(예고+근거) ③이렇게 수정해서 성과가 이렇게 됐다(결과) ④수정했는데 성과가 별로라 이번엔 이렇게(반성+새 계획) ⑤위를 한 보고에 조합. 전부 "조치+당시 지표" 구조가 있어야 생성 가능 — 이 구조가 이력 설계의 존재 이유.

## 8. 에러 처리

`friendly-error` 패턴 유지 — "로그인이 만료됐어요. 다시 로그인해 주세요" 같은 일상 한글. 영문 기술용어 금지(루트 CLAUDE.md).

## 9. 테스트

- 순수 로직(마이그레이션 판정, 잠금 상태 판정, 이력→facts 변환)은 vitest.
- 로그인·승인·관리자 흐름은 테스트 계정 2개(관리자/일반)로 수동 QA 체크리스트.

## 10. 단계

- **1단계 (이번 계획)**: 가입·로그인·승인·잠금 + 테이블 4개 + 마이그레이션 + 관리자 탭 + brief_token 폐기.
- **2단계 (1단계 실사용 후)**: `brief_history` + 지난 조치 추적 후보 + 보고 유형 카탈로그 반영.

순서 강제: 이력은 회원 체계 위에서만 돌아간다. 한 계획으로 합치면 20 Task를 넘어 나눴다.
