# 디브이 애드 매니저 MVP PRD

## 🎯 핵심 정보

**목적**: 네이버 광고 대시보드(`ads.naver.com`)에서 파워링크 키워드의 순위/예상 입찰가를 실시간으로 옆에 띄우고 1-클릭 입찰가 변경까지 처리해 입찰 의사결정·실행 시간을 단축한다.
**사용자**: 다수의 광고주 계정을 동시에 운영하는 **네이버 광고 대행사 AE**가 1차 타겟. 인하우스 광고 운영자, 브랜드 스토어 셀러도 포함.

> 본 프로젝트는 Chrome MV3 확장으로 배포된다. 일반 웹 앱이 아니라 호스트 페이지(`ads.naver.com`)에 콘텐츠 스크립트를 주입하는 구조이며, MVP에서는 **별도의 회원 인증/라이선스 검증 없이** 검색광고 API 자격증명만 등록되면 기능이 활성된다. 검색광고 API의 입찰가/볼륨 응답은 **시장 단위 추정치**이므로 호출자 customerId와 무관하게 동일하다 — 따라서 자격증명은 **`customerId` + `accessLicense` + `secretKey` 1쌍만 등록**해 두면 사용자가 보고 있는 광고주가 누구든 동일하게 동작한다. 광고주 매칭·다중 자격증명 개념은 사용하지 않는다.
>
> **⏸️ F002/F003 보류 (2026-05-19)**: 본 PRD의 F002 쇼핑검색광고 그룹 뷰 / F003 소재 상세 패널은 MVP 범위에서 제외됨. 사유: ① 쇼핑검색광고는 검색광고 API에 전용 estimate endpoint가 없어 "키워드별 1~10위 시장가" 자체가 의미 약함(파워링크 시장가만 가능). ② 키워드 단위 평균순위는 광고관리자 자체도 제공하지 않음(소재 단위만). ③ 광고관리자 "제외키워드 추가" 모달이 이미 키워드별 노출/클릭/비용/전환율을 정렬·1-클릭 추가까지 충분히 잘 제공함 — 우리의 inline 펼침이 추가 가치를 못 줌. 다음 방향(ROI 임계값 자동 분류 + 일괄 액션)은 별도 brainstorming에서 새 기능 ID로 재정의 예정. Spike B 결과(`admng_exp_keyword` / `ad-account v2` 비공식 endpoint)는 추후 재사용 가능하므로 메모리 `project_spike_b_shopping_endpoints`에 보존.

## 🚶 사용자 여정

```
1. Chrome 웹스토어에서 확장 설치
   ↓ 설치 직후
2. 옵션 페이지 (자동 또는 사용자 직접 진입)
   ↓ 검색광고 API 자격증명(customerId + accessLicense + secretKey) 1쌍 등록
3. ads.naver.com 광고 대시보드 방문
   ↓ 콘텐츠 스크립트가 URL 패턴에 따라 화면 종류 감지
   [자격증명 등록]   → 파워링크 오버레이 활성 (시장 단위 입찰가/볼륨)
   [자격증명 미등록] → "검색광고 API 자격증명 미등록" 안내 + 옵션 페이지 바로가기
   ↓ URL 패턴에 따라 분기
   [파워링크 키워드 화면] → 콘텐츠 오버레이 — 파워링크 키워드 테이블
   (쇼핑검색광고 화면은 F002/F003 보류로 비활성)
4. (필요 시) 팝업 페이지 진입 → 자격증명 상태 확인, 캐시 강제 갱신
5. (자격증명 수정 필요 시) 옵션 페이지 재진입
```

## 🗄️ 데이터 모델

본 확장은 서버 DB를 보유하지 않는다. 모든 사용자 데이터는 `chrome.storage.local`에만 보관된다.

**데이터 전송 원칙 (2026-07-16 개정)**: 광고 데이터는 **네이버와 사내 서버(Supabase Edge Function) 외로 나가지 않는다.** F-Brief(보고 문구) AI 문장 생성 시 AE가 체크한 사실(facts)만 사내 Edge Function 1개를 경유해 Gemini로 전달되며, **서버는 어떤 데이터도 저장·로깅하지 않는다** (에러 로그도 상태코드만). 리포트 전체를 보내는 일은 없다.

### SearchadCredentials (chrome.storage.local 키: `searchadCredentials`)
> 자격증명 1쌍. `src/shared/searchad.ts`의 `loadCredentials`/`saveCredentials`/`clearCredentials`로 관리.

| 필드 | 설명 | 타입 |
|------|------|------|
| customerId | 네이버 검색광고 광고주 ID (HMAC `X-Customer` 헤더용) | string |
| accessLicense | API 액세스 라이선스 | string |
| secretKey | API 시크릿 키 (HMAC 서명용) | string |

### KeywordVolumeCache (chrome.storage.local 키: `volume_cache:<keyword>`)
> 검색광고 API 응답이 시장 단위 추정치라 호출자 customerId와 무관하게 동일 — 키워드 단위로만 스코프.

| 필드 | 설명 | 타입 |
|------|------|------|
| keyword | 정규화된 키워드 | string |
| rank_to_bid | 순위(1~15) → 예상 입찰가 매핑. API 단일 호출 max에 맞춰 `MAX_POSITION` 상수로 일원화 | object |
