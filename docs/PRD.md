# 디브이 애드 매니저 MVP PRD

## 🎯 핵심 정보

**목적**: 네이버 광고 대시보드(`ads.naver.com`)에서 파워링크 키워드의 순위/예상 입찰가와 쇼핑검색광고 소재의 자동매칭 키워드별 현재 순위/예상 입찰가를 실시간으로 옆에 띄워 입찰 의사결정 시간을 단축한다.
**사용자**: 다수의 광고주 계정을 동시에 운영하는 **네이버 광고 대행사 AE**가 1차 타겟. 인하우스 광고 운영자, 브랜드 스토어 셀러도 포함.

> 본 프로젝트는 Chrome MV3 확장으로 배포된다. 일반 웹 앱이 아니라 호스트 페이지(`ads.naver.com`)에 콘텐츠 스크립트를 주입하는 구조이며, 회원 인증 대신 **라이선스 키 검증**(Supabase RPC) 방식을 사용한다. **사용자(대행사 AE)는 일반적으로 여러 광고주 계정의 대시보드에 브라우저 쿠키로 접근 가능하지만, 모든 계정의 검색광고 API 자격증명을 보유하지는 않는다.** 따라서 자격증명은 **광고주(customerId)별로 N개 등록**하며, 현재 활성 광고주를 자동 감지해 매칭 분기한다. 코어 라이브러리는 자매 프로젝트 `naver-tag-picker`와 공유한다.
>
> **🟡 TBD**: 쇼핑검색광고 키워드별 순위/예상 입찰가의 **데이터 소스는 미정**(검색광고 API 쇼핑 endpoint / `ads.naver.com` DOM 추출 / 내부 XHR 재호출 등). 자매 프로젝트의 `src/lib/search-popular.ts`는 스마트스토어 셀러 백오피스 API(`sell.smartstore.naver.com`)를 호출하지만, 대행사 AE 타겟에서는 셀러 계정을 보유하지 않아 부적합. 구현 단계에서 별도 결정. F002/F003의 자격증명 매칭 정책은 데이터 소스 확정 후에 정의한다.

## 🚶 사용자 여정

```
1. Chrome 웹스토어에서 확장 설치
   ↓ 설치 직후
2. 옵션 페이지 (자동 또는 사용자 직접 진입)
   ↓ 라이선스 키 입력 → Supabase RPC 검증 (베이직 tier 활성 시 전체 기능 해제)
   ↓ 운영 중인 광고주별로 검색광고 API 자격증명 N개 추가 (선택)
   ↓ 자격증명이 없는 계정도 사용은 가능 (쇼핑 기능 한정)
3. ads.naver.com 광고 대시보드 방문
   ↓ 콘텐츠 스크립트가 현재 활성 광고주 customerId를 자동 감지
   ↓ 등록된 자격증명과 매칭 (파워링크 기준)
   [매칭 성공]  → 파워링크 오버레이 활성
   [매칭 실패]  → 파워링크는 "이 계정 API 미등록" 안내
   ↓ 쇼핑 기능(F002/F003)의 매칭 정책은 데이터 소스 확정 후 결정 (TBD)
   ↓ URL 패턴에 따라 분기
   [파워링크 키워드 화면]  → 콘텐츠 오버레이 — 파워링크 키워드 테이블
   [쇼핑검색광고 그룹 화면] → 콘텐츠 오버레이 — 쇼핑검색광고 그룹 뷰
   [쇼핑검색광고 소재 화면] → 콘텐츠 오버레이 — 쇼핑검색광고 소재 상세
4. (필요 시) 팝업 페이지 진입 → 라이선스 상태, 현재 광고주 매칭 상태 확인, 캐시 강제 갱신
5. (자격증명 추가/수정 필요 시) 옵션 페이지 재진입
```

## ⚡ 기능 명세

### 1. MVP 핵심 기능

| ID | 기능명 | 설명 | MVP 필수 이유 | 관련 페이지 |
|----|--------|------|---------------|-------------|
| **F001** | 파워링크 키워드 순위·입찰가 오버레이 | 파워링크 키워드 테이블 각 행 옆에 현재 세팅된 입찰가의 추정 순위(예: "현재 4위")와 1위~10위의 예상 입찰가를 표시. 현재 광고주 customerId에 매칭되는 자격증명이 등록돼 있을 때만 활성. **데이터 소스**: 네이버 검색광고 API `POST /estimate/average-position-bid/keyword` (요청 body의 `items: [{key, position}]`로 키워드별 1~10위 입찰가 일괄 조회). 응답 schema의 정확한 필드명은 구현 1일차 실호출로 확정 | 본 확장의 첫 번째 핵심 가치. 입찰가 결정 시 광고 대시보드를 떠나지 않고 즉시 비교 가능 | 콘텐츠 오버레이 — 파워링크 키워드 테이블 |
| **F002** | 쇼핑검색광고 그룹 뷰 소재별 inline 펼침 | 그룹 페이지의 소재 행 옆 토글 버튼을 누르면 그 자리에 펼쳐지며, 소재의 자동매칭 키워드 목록과 각 키워드의 현재 순위·1위~10위 예상 입찰가를 테이블로 표시. **여러 소재를 동시에 펼쳐 비교 가능**. **데이터 소스 및 자격증명 매칭 정책은 TBD** (위 핵심 정보 참고) | 두 번째 핵심 가치. 쇼핑검색광고는 키워드를 직접 지정할 수 없어 소재별 매칭 키워드 파악 자체가 운영 난점 | 콘텐츠 오버레이 — 쇼핑검색광고 그룹 뷰 |
| **F003** | 쇼핑검색광고 소재 상세 키워드 패널 | 소재 단독 상세 페이지에서 그 소재의 자동매칭 키워드 전체 목록과 키워드별 현재 순위·1위~10위 예상 입찰가를 풀 패널로 표시. 정렬/검색 가능. **데이터 소스 및 자격증명 매칭 정책은 TBD** | 그룹 뷰의 inline 펼침과 동일 정보를 단일 소재 페이지에서는 더 넓은 영역으로 보여줘야 정독 가능 | 콘텐츠 오버레이 — 쇼핑검색광고 소재 상세 |

### 2. MVP 필수 지원 기능

| ID | 기능명 | 설명 | MVP 필수 이유 | 관련 페이지 |
|----|--------|------|---------------|-------------|
| **F010** | 라이선스 키 등록·검증 | 사용자가 입력한 라이선스 키를 Supabase RPC `verify_access`로 검증, 결과(tier·만료일)를 5분 TTL로 캐시. MVP에서는 **베이직 tier 단일 등급**이며 활성이면 모든 핵심 기능 사용 가능. 미검증·만료 시 모든 핵심 기능 비활성. 디바이스 ID와 키만 외부 전송 | 회원가입/로그인을 대체하는 본 확장의 유일한 인증·과금 게이트 | 옵션 페이지 |
| **F011** | 검색광고 API 자격증명 다중 관리 | 광고주(customerId)별로 `accessLicense`·`secretKey`를 N개 등록·수정·삭제. 별칭(예: "A고객사", "B브랜드")을 함께 저장해 식별성 강화. 비밀값은 `chrome.storage.local`에만 저장, 외부 전송 X. 대행사 AE가 운영하는 여러 광고주를 한 확장에서 모두 다룰 수 있게 함. **마이그레이션**: 현재 코드(`searchad.ts`)는 단일 자격증명(`searchadCredentials` 단일 키)만 지원. 신규 키(`searchad_credentials` 배열)로 1회 마이그레이션(구 키 존재 시 `[{...단일자, label: "기본"}]`로 변환 후 삭제). naver-tag-picker 코어 동기화는 별도 작업으로 분리 — 본 repo는 어댑터 함수(`loadCredentials()` → `loadAllCredentials()[0]`)로 호환 유지 | 대행사 AE 워크플로우의 핵심. 일부 광고주의 API 자격증명만 보유한 상태도 흔하므로 매칭되지 않는 광고주는 자연스럽게 기능 비활성으로 폴백 | 옵션 페이지 |
| **F012** | 라이선스 상태 표시 및 캐시 강제 갱신 | 팝업에서 라이선스 tier·만료일·검증 시각, 그리고 현재 활성 탭 정보를 확인. 활성 탭에 해당하는 customerId-scoped 캐시만 부분 만료 후 재조회 트리거 (전체 캐시 클리어 X). 활성 탭 식별은 `chrome.tabs.query({active:true, currentWindow:true})` 사용 — host_permissions만으로 충분한지 구현 첫날 1회 확인, 필요 시 `permissions`에 `"activeTab"` 추가 | 캐시가 오래된 값을 보여줄 때 사용자가 즉시 새 데이터로 갱신할 수 있어야 함 | 팝업 페이지 |
| **F013** | 활성 광고주 자동 감지·매칭 표시 | 콘텐츠 스크립트가 `ads.naver.com` DOM/URL에서 현재 활성 광고주 customerId(또는 라벨)를 추출, 등록된 자격증명 목록과 매칭. 매칭 결과를 콘텐츠 오버레이(미매칭 시 안내 배지)와 팝업(현재 광고주 + 자격증명 상태)에서 일관되게 노출. **감지 전략은 Spike A로 확정**(부록 참조) — URL 패턴 → DOM 헤더 + MutationObserver → 사용자 명시 드롭다운 폴백의 계단식 채택. 자동 감지 실패 시에도 폴백 드롭다운으로 항상 매칭 가능해야 함 | 자격증명을 가진 광고주와 가지지 않은 광고주를 오가는 대행사 AE에게 "지금 어떤 계정을 보고 있는지 / 기능이 왜 비활성인지"를 즉시 알려줘야 혼란 없음 | 콘텐츠 오버레이 — 파워링크 키워드 테이블, 콘텐츠 오버레이 — 쇼핑검색광고 그룹 뷰, 콘텐츠 오버레이 — 쇼핑검색광고 소재 상세, 팝업 페이지 |

### 3. MVP 이후 기능 (제외)

- 키워드별 입찰가 변동 히스토리·차트
- 경쟁사 광고 분석
- 자동 입찰가 제안·일괄 변경
- 광고주별 일·주·월 단위 통계 요약 대시보드
- 슬랙·이메일 알림

## 📄 페이지별 상세 기능

### 콘텐츠 오버레이 — 파워링크 키워드 테이블

> **구현 기능:** `F001`, `F013` | **UI 위치:** `ads.naver.com` 파워링크 키워드 화면에 자동 주입

| 항목 | 내용 |
|------|------|
| **역할** | 파워링크 키워드 운영 화면에서 입찰가 결정에 필요한 순위·예상 입찰가 정보를 키워드 행 바로 옆에 inline 표시. 현재 광고주의 자격증명 등록 여부를 한눈에 인식 가능하게 함 |
| **진입 경로** | 사용자가 `ads.naver.com`의 파워링크 키워드 관리 화면에 진입하면 콘텐츠 스크립트가 URL 패턴을 감지하여 자동 활성화 |
| **사용자 행동** | 키워드 행을 훑으며 "현재 입찰가는 몇 위인지", "1위가 되려면 얼마를 더 써야 하는지"를 즉시 비교. 활성 광고주가 미등록 상태면 옵션 페이지에서 자격증명을 추가하고 돌아옴 |
| **주요 기능** | • 현재 광고주 customerId 자동 감지 → 등록된 자격증명과 매칭 (F013)<br>• 매칭 성공 시 키워드 행 옆에 **현재 입찰가의 추정 순위** 배지 표시 (예: "현재 4위")<br>• 행 펼침 시 **1위~10위의 예상 입찰가** 테이블 표시<br>• 매칭 실패 시 "이 광고주의 검색광고 API 자격증명 미등록" 안내 배지 + 옵션 페이지 바로가기<br>• 라이선스 미검증/만료 시 잠금 상태 표시<br>• 검색광고 API 429/401/네트워크 에러 시 친화적 메시지 표시 |
| **다음 이동** | 동일 페이지 내 키워드 추가/수정/삭제 시 오버레이 자동 갱신. 페이지 이탈 시 콘텐츠 스크립트 정리. 매칭 실패 안내 클릭 시 옵션 페이지 |

> **구현 기능:** `F002`, `F013` | **UI 위치:** `ads.naver.com` 쇼핑검색광고 그룹 관리 화면에 자동 주입

## 🗄️ 데이터 모델

본 확장은 서버 DB를 보유하지 않는다. 사용자 측 영속 저장소는 `chrome.storage.local`이며, 라이선스 검증만 Supabase의 기존 RPC를 호출한다.

> **🔄 마이그레이션 노트**: 현재 코드의 `volume-cache.ts`는 `volumeCache` 단일 평면 객체 구조다. 본 PRD의 customerId-scoped 키 스킴(`volume_cache:<customer_id>:<keyword>` 등)으로 변경 시 1회 마이그레이션 필요 — 구 캐시는 첫 부팅 시 일괄 폐기(매칭 불가)하거나 customerId="legacy"로 일시 보존 후 TTL 만료 대기. 자매 프로젝트 `naver-tag-picker`의 캐시 사용처와 충돌 방지 위해 키 prefix를 `dvads:`로 추가하는 것도 옵션.
